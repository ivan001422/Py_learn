### Обработка исключений (try/except)

В Python ошибки — это не просто сбои, а полноценный **инструмент управления потоком**. В отличие от многих других языков, где ошибки стараются предусмотреть заранее с помощью проверок (подход **LBYL** — *Look Before You Leap*), Python поощряет подход **EAFP** (*Easier to Ask for Forgiveness than Permission* — проще попросить прощения, чем разрешения). Это означает, что мы **сначала пытаемся выполнить действие**, и только если что-то идёт не так, перехватываем исключение и обрабатываем его.


### EAFP vs LBYL

В Python принят подход **EAFP** (*Easier to Ask for Forgiveness than Permission* — проще попросить прощения, чем разрешения). Это означает, что мы пытаемся выполнить действие, а если возникает исключение, обрабатываем его. Альтернатива – **LBYL** (*Look Before You Leap* — посмотри, прежде чем прыгнуть), то есть сначала проверяем условия, потом выполняем.

Пример для деления:

- LBYL: `if divisor != 0: result = 100 / divisor`
- EAFP: `try: result = 100 / divisor except ZeroDivisionError: ...`

**Почему EAFP предпочтительнее в Python?**

- **Производительность** – в случае успеха блок `try` работает быстрее, чем проверка `if` (особенно если проверка сложная).
- **Безопасность** – в многопоточном окружении между проверкой и действием состояние может измениться (гонка). Исключение перехватывается атомарно.
- **Читаемость** – код часто становится короче и понятнее, особенно если нужно выполнить много операций с возможными ошибками.

Однако не стоит злоупотреблять EAFP в горячих циклах, где исключения могут возникать часто – создание и перехват исключений дорого. Если ошибка ожидаема (например, часто отсутствуют ключи в словаре), лучше использовать `dict.get()` или проверку `if key in dict`.

### Когда нужно обрабатывать ошибки (и когда НЕ нужно)

**Золотое правило:** обрабатывайте исключение **только на том уровне**, где вы можете с ним что-то сделать — повторить операцию, предложить альтернативу, записать лог и продолжить работу, показать понятное сообщение пользователю. Не ловите ошибки просто так, «чтобы программа не упала» — это скрывает реальные проблемы и усложняет отладку.

**Когда ОБЯЗАТЕЛЬНО обрабатывать:**

- **Взаимодействие с пользователем** (ввод данных, GUI, консоль) — чтобы показать понятное сообщение вместо страшного стек-трейса.
- **Внешние ресурсы** (файлы, сеть, базы данных) — чтобы корректно закрыть соединения, повторить запрос, переключиться на резервный сервер.
- **Преобразование типов** (например, ввод пользователя → число) — чтобы переспросить или предложить исправить ввод.
- **Логика с повторными попытками** (retry-механизмы) — при временных сбоях.

**Когда НЕ нужно обрабатывать (лучше дать упасть):**

- **Программные ошибки** (`IndexError`, `AttributeError`, `TypeError`) — это баги в коде, их надо исправлять, а не маскировать.
- **Ситуации, которые нельзя исправить на этом уровне** (нет прав доступа, диск полон) — можно только передать выше, чтобы на верхнем уровне завершить программу или сообщить администратору.

### Синтаксис

Базовый блок выглядит так:

```python
try:
    # потенциально опасный код
except SomeException:  # название ошибки
    # обработка ошибки
```

Можно указать несколько типов исключений в одном `except`:

```python
except (ZeroDivisionError, TypeError):
    # общий обработчик для обоих типов
```

Для получения объекта исключения (чтобы, например, прочитать его сообщение) используется ключевое слово `as`:

```python
except ValueError as ex:
    print(f"Ошибка: {ex.args}")
```

### Иерархия исключений

Все исключения в Python являются экземплярами классов, которые наследуются от `BaseException`. От него напрямую наследуются:

- `Exception` – базовый класс для всех **обычных** ошибок (именно их мы обычно перехватываем).
- `SystemExit` – вызывается при завершении программы через `sys.exit()`.
- `KeyboardInterrupt` – вызывается при нажатии Ctrl+C.

Важно: `SystemExit` и `KeyboardInterrupt` **не являются** подклассами `Exception`. Это сделано специально, чтобы блок `except Exception:` не перехватывал их случайно. Если бы они были подклассами `Exception`, то любой код с `except Exception` перехватил бы и запрос на завершение программы, что помешало бы корректному выходу.

**Правило:** никогда не используйте пустой `except:` (без указания типа). Он перехватывает абсолютно всё, включая `SystemExit` и `KeyboardInterrupt`, что может сделать программу неуправляемой (вы не сможете завершить её по Ctrl+C). Вместо этого всегда указывайте либо конкретные типы, либо `except Exception:` (если нужно перехватывать все обычные ошибки).

Можно использовать несколько блоков `except` для разных типов:

```python
try:
    ...
except ZeroDivisionError:
    print("Деление на ноль")
except ValueError:
    print("Неверное значение")
except Exception as e:
    print(f"Неизвестная ошибка: {e}")
```

**Важно:** порядок блоков имеет значение. Python проверяет их сверху вниз и останавливается на первом совпавшем типе. Если сначала указать родительский класс (`Exception`), то он перехватит все ошибки, и более конкретные блоки (`ZeroDivisionError`, `ValueError`) никогда не сработают. Поэтому **сначала пишем дочерние (конкретные) исключения, затем родительские (общие)**.

### Блоки else и finally

- **`else`** – выполняется, если блок `try` завершился без исключений. Это позволяет отделить «нормальный» код от кода обработки ошибок.
- **`finally`** – выполняется **всегда**, независимо от того, возникло исключение или нет. Используется для освобождения ресурсов (закрытие файлов, соединений, сокетов).

Пример:

```python
try:
    f = open("file.txt")
    data = f.read()
except IOError:
    print("Ошибка чтения")
else:
    print("Файл прочитан успешно")
finally:
    f.close()   # всегда закрываем файл
```

Блок `finally` выполняется даже после `return` внутри `try` или `except`. Это гарантирует, что ресурсы будут освобождены. Однако в современном Python для работы с ресурсами (файлы, сетевые соединения) чаще используют **контекстные менеджеры** (конструкция `with`), которые автоматически вызывают `__exit__` и гарантируют освобождение ресурсов. Например, `with open("file.txt") as f:` автоматически закроет файл, даже если произойдёт ошибка.

### Создание собственных исключений

Часто встроенных исключений недостаточно, потому что они не несут **предметного смысла**. Например, `ValueError` слишком общий – он не отличает «недостаточно средств» от «неверный формат номера счёта». Создавая своё исключение, вы даёте ему осмысленное имя и можете передавать дополнительную информацию (баланс, запрошенную сумму, код ошибки и т.д.).

Для создания собственного исключения достаточно унаследовать от `Exception` (или от подходящего подкласса). Можно переопределить `__init__`, чтобы принимать нужные параметры, и добавить методы для удобства.

Прежде чем рассматривать конкретный пример, разберём, как устроен базовый класс `Exception`. Его конструктор выглядит примерно так (упрощённо):

```python
class Exception:
    def __init__(self, *args):
        self.args = args   # сохраняет все переданные аргументы как кортеж
```

Конструктор `Exception` принимает неограниченное количество аргументов и сохраняет их в кортеже `args`. Это сделано для того, чтобы обеспечить правильный вывод информации при печати ошибки:

```python
def __str__(self):
    if len(self.args) == 1:
        return str(self.args[0])   # если один аргумент — выводим его как есть
    else:
        return str(self.args)      # если несколько — выводим кортеж
```

Таким образом, если вы передаёте в `super().__init__()` **одну строку**, то при выводе ошибки вы увидите именно эту строку. Если передать несколько аргументов по отдельности, то выведется кортеж — обычно это некрасиво. Поэтому чаще всего формируют одно сообщение, включающее все нужные данные.

Пример:

```python
class MyError(Exception):
    def __init__(self, message, number_error):
        # передаём в родительский класс одну строку
        super().__init__(f'{message} number {number_error}')

raise MyError("Error", 123)  # выведет "Error number 123"
```

Если бы мы написали `super().__init__(message, number_error)`, то при печати увидели бы `('Error', 123)`.

**Преимущества создания собственных исключений:**

- **Ясность** – имя исключения сразу говорит о проблеме.
- **Контекст** – обработчик может получить доступ к балансу, запрошенной сумме и другим данным.
- **Гибкость** – можно добавить методы для вычислений или логирования.

При проектировании библиотек или API собственные исключения делают интерфейс более понятным и удобным для клиентского кода.

#### Как правильно создавать собственные исключения (архитектура)

1. Создайте **базовое исключение** для всего вашего модуля (или всего приложения):

```python
# exceptions.py
class MyAppError(Exception):
    """Базовое исключение для всех ошибок приложения."""
    pass
```

2. Затем от него наследуйте конкретные ошибки:

```python
class ValidationError(MyAppError):
    """Ошибка валидации входных данных."""
    def __init__(self, field, value, message):
        self.field = field
        self.value = value
        self.message = message
        super().__init__(f"Поле '{field}': {message} (значение: {value})")

class DatabaseError(MyAppError):
    """Ошибка при работе с базой данных."""
    pass

class InsufficientFunds(MyAppError):
    """Ошибка недостаточности средств."""
    def __init__(self, balance, amount):
        self.balance = balance
        self.amount = amount
        shortage = amount - balance
        super().__init__(f"Недостаточно средств. Баланс: {balance}, запрошено: {amount}")
        self.shortage = shortage
```

**Что должно быть в конструкторе и как вызывать `super()`:**

- Всегда вызывайте `super().__init__(сообщение)` — это гарантирует, что сообщение попадёт в `args` и будет выводиться при печати.
- Добавляйте нужные атрибуты (например, `field`, `value`, `balance`, `amount`), чтобы в блоке `except` можно было их извлечь и принять решение.
- Формируйте **читаемое сообщение** из переданных данных, чтобы оно было понятно пользователю (и в логах).

### Как правильно вызывать собственные исключения?

Существует два основных способа.

#### 1. Явный вызов через `raise` в блоке `try` (или прямо в коде)

Вы сами проверяете условие и, если оно нарушено, выбрасываете своё исключение.

Пример на банковском приложении:

```python
class AmountError(Exception):
    def __init__(self):
        super().__init__("Сумма не может быть меньше или равна нулю")

class BankAccount:
    def __init__(self, balance: int = 0):
        self.balance = balance

    def popolnenie(self, amount: int) -> None:
        if amount <= 0:
            raise AmountError   # явно вызываем нашу ошибку
        self.balance += amount

Ivan = BankAccount(100)
Ivan.popolnenie(-10)   # выбросит AmountError
```

Здесь для вызова нашей ошибки мы используем инструкцию `if` — это абсолютно нормально и является хорошим тоном. Иногда вы можете захотеть передать в исключение динамические данные, например:

```python
class AmountError(Exception):
    def __init__(self, amount):
        super().__init__(f"Недопустимая сумма: {amount}")

# ...
if amount <= 0:
    raise AmountError(amount)
```

#### 2. «Обёртывание» (wrapping) базовой ошибки в свою

Представьте, что у вас есть функция, которая вычисляет среднее арифметическое списка чисел. Если список пуст, при вычислении произойдёт `ZeroDivisionError`. Но для пользователя эта ошибка не имеет смысла — он ожидает, что ему скажут «список пуст». В этом случае вы ловите стандартное исключение и бросаете своё.

```python
class EmptyListError(Exception):
    """Выбрасывается, когда передан пустой список для вычислений."""
    pass

def average(numbers):
    try:
        return sum(numbers) / len(numbers)
    except ZeroDivisionError:
        # Ловим стандартную ошибку и бросаем свою
        raise EmptyListError("Нельзя вычислить среднее для пустого списка")
```

Теперь, если вызвать `average([])`, программа выбросит `EmptyListError` с понятным сообщением.

**Важное дополнение:** если вы хотите сохранить исходную ошибку в цепочке (для отладки), используйте синтаксис `raise ... from ...`:

```python
except ZeroDivisionError as e:
    raise EmptyListError("Нельзя вычислить среднее") from e
```

Тогда в выводе вы увидите обе ошибки, и будет ясно, что исходной причиной был `ZeroDivisionError`. Если вы хотите скрыть низкоуровневую причину (например, чтобы не раскрывать детали реализации), используйте `from None`:

```python
except ZeroDivisionError:
    raise EmptyListError("Нельзя вычислить среднее") from None
```

В этом случае исходная ошибка не будет показана, и пользователь увидит только ваше сообщение.
